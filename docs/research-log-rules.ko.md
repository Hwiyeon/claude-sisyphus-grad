# 연구 로그 작성 규칙

연구 로그는 나와 Claude 사이의 공유 기억입니다. 과정을 생략한 로그는 로그가 없는 것보다 나쁩니다 — 잘못된 안도감을 만들기 때문입니다. 이 규칙들은 로그에서 가장 유용한 정보가 대부분 "잘 안 됐던 것"이라는 사실에서 나왔습니다.

---

## 핵심 원칙

### 1. 과정을 생략하지 않는다

로그는 성공 보고서가 아닙니다. 막힌 길을 포함해 시도한 모든 것의 완전한 기록입니다.

learning rate 세 값을 시도해서 둘이 실패했다면, 셋 모두 실패 사유와 함께 로그에 들어갑니다. 아키텍처 변경을 논의하고 채택하지 않기로 했다면, 그 논의도 로그에 들어갑니다.

**이유:** 미래의 세션 — 본인 것도 포함 — 은 기록되지 않은 아이디어를 다시 탐색합니다. 문서화된 실패는 차단된 탐색 경로이고, 문서화되지 않은 실패는 반복되는 작업입니다.

### 2. 기각된 아이디어도 사유와 함께 기록한다

검토했지만 채택하지 않은 모든 아이디어는 **■ The 'Missing' Details** 섹션에 구체적인 기각 사유와 함께 기록해야 합니다.

좋은 예:
> ALiBi 검토했으나 기각 — subset에서 val_acc +1.2% 개선이지만 340ms/batch로 RoPE(180ms/batch) 대비 지연시간 예산 초과.

너무 막연한 예:
> ALiBi는 잘 안 됐음.

**이유:** "이걸 시도했다"는 "왜 멈췄는지"가 없으면 유용하지 않습니다. 이유가 정보입니다.

### 3. 느낌이 아닌 수치를 인용한다

모델 동작에 대한 모든 관찰은 구체적인 메트릭 값에 근거해야 합니다.

좋은 예:
> val_loss가 epoch 12–18 걸쳐 0.43에서 정체 (Δ < 0.002); 학습률 1e-3 → 5e-4 감소도 가시적 효과 없음.

허용 불가:
> loss가 높았고 별로 개선이 없었음.

### 4. 컨텍스트 제로인 미래 독자를 위해 쓴다

다음 세션이 이 대화에 대한 기억이 전혀 없다고 가정합니다. 로그만 읽고 아래 내용을 복원할 수 있어야 합니다:
- 현재 아키텍처가 무엇인가
- 무엇을 시도했고 왜 포기했는가
- 열린 질문이 무엇인가
- 다음에 무엇을 해야 하는가

미래의 Claude가 (또는 3주 후의 본인이) 로그만 읽고 정확히 멈춘 곳에서 이어갈 수 없다면, 로그는 불완전한 것입니다.

### 5. 압축은 적이다

상세 내용이 있는데 요약하지 마세요. "데이터 파이프라인을 개선했다"고 쓰지 말고 구체적으로 무엇이 바뀌었는지, 개선 전후 메트릭이 어떻게 됐는지, 왜 그 변경을 했는지를 쓰세요.

예외: 타이포 수정, 파일 경로 오류 해결 같은 사소한 디버깅 과정은 로그할 필요 없습니다.

## 섹션별 작성 규칙

### ■ Current Objectives

막연한 목표가 아닌 오늘 다루는 **구체적인 병목**을 기술합니다.

좋은 예:
> train_loss는 계속 감소하는데 epoch 8 이후 val_loss가 발산하는 원인 파악. 가설: 정규화 부족 또는 데이터 augmentation 불충분으로 인한 overfitting.

유용하지 않은 예:
> 모델 계속 개선.

---

### ■ The 'Missing' Details ← 가장 중요한 섹션

이 섹션은 필수이며 축약할 수 없습니다. 포함 내용:

- **기각된 아이디어** — 논의했지만 채택하지 않은 아이디어, 구체적 기각 사유 포함
- **배제한 변수** — 명시적으로 시도하지 않기로 한 파라미터나 접근법, 기술적 근거 포함
- **실패한 시도** — 시도했지만 안 된 것, 실패 양상 기술
- **미해결 의문** — 세션 후에도 남아있는 열린 질문

이 섹션이 비어있거나 항목이 하나뿐이라면, 세션이 극도로 단순했거나 로그가 불완전한 것입니다.

---

### ■ Confirmed Decisions

결정은 구체적이어야 합니다. 실제 값, 수식, 또는 확정된 하드 제약을 포함합니다.

좋은 예:
> 확정: hidden_dim=512, num_heads=8, dropout=0.1. 근거: {256, 512, 1024} 스윕 결과 512가 val_acc 87.3%로 최적; 1024는 subset에서 epoch 5 이후 overfitting 발생.

유용하지 않은 예:
> 모델 아키텍처를 결정했음.

---

### ■ Architecture Update

구조적 변경이 있을 때마다 필수입니다. 항상 ASCII 다이어그램을 포함합니다.

```
Before:
Input → Embedding(512) → TransformerEncoder(6L) → CLS → Linear → Output

After:
Input → Embedding(512) → TransformerEncoder(6L) → [CLS, Mean-pool] → Concat → Linear(1024→num_classes) → Output
```

이번 세션에서 아키텍처 변경이 없었다면 이 섹션은 생략 가능합니다.

---

### ■ Technical Artifacts

- 수식: KaTeX 블록 (`$$...$$` 블록, `$...$` 인라인)
- 코드: 언어 태그가 있는 fenced 코드 블록
- 자명하지 않은 것, 또는 핵심 결정을 담은 아티팩트만 포함

---

### ■ Experiment Results

이미지는 **로그 파일과 같은 폴더**에 저장하고 상대경로로 참조합니다:

```markdown
![val loss curve](val_loss_epoch.png)
```

포함 내용:
- 시각화가 보여주는 것
- 핵심 관찰
- 가설을 확인하는지 반박하는지

---

### ■ Future Directions

이 섹션은 **실행 가능**해야 합니다. 다음 세션의 시작점입니다.

좋은 예:
> 1. ALiBi를 max_position=2048로 subset에서 테스트 (오늘 시간 부족으로 미룸)
> 2. epoch 8 발산 원인 파악을 위해 gradient norm 로깅 추가 후 재실험
> 3. subset val_acc > 85% 달성 시 full dataset 실험으로 진행

유용하지 않은 예:
> 실험 계속. 더 시도해봄.

## 2단계 로깅 규칙

연구 디렉토리는 두 단계의 로그를 사용합니다. 어느 단계에 무엇을 써야 하는지 이해하는 것이 중요합니다.

**일별 로그** (`logs/YYYY-MM-DD/log.md`):
- 당일 모든 주제에 걸쳐 일어난 일의 인덱스
- 여러 주제에 걸친 관찰과 결정
- 특정 주제에 속하지 않는 실험 결과
- 여러 주제를 건드린 세션은 각 주제의 상세 로그를 가리키는 daily entry 작성

**토픽 로그** (`topics/<주제>/log.md`):
- 해당 모듈의 primary 상세 기록
- ASCII 다이어그램, 수식, 코드 스니펫, 전체 시도 이력
- 특정 주제 발전 과정의 권위 있는 기록

**규칙:** 중복하지 않습니다. 토픽 로그에 속하는 내용은 거기에 쓰고, 일별 로그에는 한 줄 참조만 남깁니다. 여러 주제에 걸친 내용은 일별 로그에 씁니다.

## 아키텍처 파일 vs. 실험 로그 구분

### `architecture_<주제>.md` — 현재 best 명세만

이 파일은 "현재 아키텍처가 무엇인가?"에 답합니다.

- **현재 best 명세만** 포함
- 갱신 시 **항상 덮어씁니다** — 파일 내에 버전을 누적하지 않음
- 변경 이력, 히스토리, before/after 비교 없음
- 현재 상태를 완전히 명세하는 데 필요한 만큼만 작성

### `experiment_log.md` — 전체 변경 이력

이 파일은 "아키텍처가 어떻게 발전했는가?"에 답합니다.

구조:
```
## 아키텍처 v3 (YYYY-MM-DD)
[v3 전체 명세]

### 알고리즘 변경: ALiBi 추가 (YYYY-MM-DD)
[무엇이 바뀌었고 왜]

#### Config 조정: lr 1e-3 → 5e-4 (YYYY-MM-DD)
[결과]
```

헤딩 레벨: `##` 아키텍처 버전 → `###` 알고리즘 변경 → `####` config 튜닝.

## `research/README.md` 갱신 규칙

README는 GitHub에 표시되는 연구 현황 요약입니다. 항상 최신 상태를 유지해야 합니다.

**갱신 시점:**
- 중요한 결정이 확정될 때
- 아키텍처가 변경될 때
- 주제의 방향이 바뀔 때
- 주간 요약을 작성할 때

**포함 내용:**
- 현재 연구 방향과 어떤 문제를 풀고 있는지
- 최근 진행 상황 (최근 1–2 세션을 2–3 문장으로)
- 핵심 결정 (가장 중요한 확정된 선택들)
- 열린 질문 (다음 세션을 이끄는 현재 미지수)

---

## 동기화 규칙

중요한 로그 업데이트 후:
```bash
git add -A && git commit -m "research: YYYY-MM-DD 로그" && git push
```

push되지 않은 로그는 다른 환경에서 실행되는 미래 세션에서 보이지 않습니다.
