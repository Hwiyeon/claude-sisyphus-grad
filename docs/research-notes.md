# Research Notes System

> Referenced from [README.md](../README.md)

The research notes system is a structured directory for documenting every experiment run, architecture decision, and discussion thread. The goal: any question about the current state or history of the research should be answerable in under 30 seconds.

---

## Directory structure

```
research/
├── CLAUDE.md                      ← operating rules (auto-loaded at session start)
├── README.md                      ← research status summary (shown on GitHub)
├── logs/
│   ├── YYYY-MM-DD/
│   │   ├── log.md                 ← daily index + non-topic details
│   │   └── *.png / *.jpg          ← experiment visualizations (same folder)
│   └── archive/
│       └── MMDD~MMDD/             ← weekly archive (moved from logs/ each Wednesday)
├── summaries/
│   └── week-MMDD~MMDD.md          ← weekly summary (generated on request)
├── topics/
│   └── <topic>/
│       ├── architecture_<topic>.md  ← current best spec only
│       ├── experiment_log.md        ← architecture changelog (newest first)
│       ├── log.md                   ← topic-specific accumulated log
│       └── discussion/              ← saved /discuss sessions
└── related_work/                  ← referenced papers and techniques
```

---

## File types and purposes

### `research/README.md`

The live research status summary. Shown directly on GitHub. Updated whenever an important decision is made or the research direction shifts.

Contents:
- Current research direction and problem statement
- Recent progress (last 1–2 sessions, 2–3 sentences)
- Key confirmed decisions
- Open questions driving the next session

### `logs/YYYY-MM-DD/log.md`

Daily index log. Written at the end of each working session.

Used for:
- Cross-topic observations from the day
- Decisions that span multiple topics
- Pointers to topic-specific logs for details
- Non-topic experiment results

Does **not** contain: detailed per-topic breakdowns — those live in `topics/<topic>/log.md`.

### `topics/<topic>/log.md`

The primary detailed record for a specific module or research thread. This is where the full trial history lives: what was tried, what failed, ASCII diagrams, formulas, and code.

Format:
```markdown
# <Module> — Progress Log

## YYYY-MM-DD
### Topic / one-line summary
- Discussion content, decisions, results
- Rejected ideas and reasons (when applicable)
- Relevant metrics, formulas, code snippets
```

### `architecture_<topic>.md`

Current best specification only. Not a changelog.

- Always overwrite on update — never append versions inside this file
- Contains only the current architecture, config, and constraints
- Answers: "What is the current version of X?"

### `experiment_log.md`

Full architecture changelog for a topic.

Structure:
```
## Architecture v3 (YYYY-MM-DD)
[full v3 spec]

### Algorithm change: added ALiBi (YYYY-MM-DD)
### Config tuning: lr 1e-3 → 5e-4 (YYYY-MM-DD)
```

Heading levels: `##` arch versions → `###` algorithm changes → `####` config tuning.

### `topics/<topic>/discussion/`

Saved `/discuss` sessions. Each file is a timestamped markdown export of a research discussion thread. Future `/discuss` sessions on the same topic reference these automatically.

### `summaries/week-MMDD~MMDD.md`

Generated by the `weekly-summary` skill. Contains a structured summary of the week's experiments, decisions, and open questions. Generated on request ("주간 요약", "weekly summary"). After generation, the corresponding daily folders are moved to `logs/archive/`.

### `related_work/`

Papers, techniques, and external references. No fixed format — typically organized by topic or date encountered.

## Two-level logging in practice

The system separates the **daily index** (what happened across topics today) from the **topic primary record** (the authoritative detail for a given module).

**Write in daily log when:**
- The observation or decision spans multiple topics
- You're recording that something happened without needing full detail (with a pointer to the topic log)
- It's a session-level summary

**Write in topic log when:**
- Detailed trial-and-error history for a specific module
- Architecture diagrams or formulas for a topic
- The authoritative record of what the current approach is

**Rule:** If in doubt, write in the topic log and leave a one-line pointer in the daily log. Never duplicate detailed content between both.

## GitHub auto-sync

The pipeline includes a PostToolUse hook that automatically pushes `research/` to GitHub every time Claude writes or edits a file inside it — no manual commit needed.

### How it works

`sync-research-log.sh` fires on every `Write` and `Edit` tool call. When the modified file path contains `/research/`, it runs in the background:

```bash
git pull --rebase origin main   # pull first to avoid conflicts
git add -A
git commit -m "research log sync: YYYY-MM-DD HH:MM"
git push origin main
```

`flock` prevents concurrent syncs when multiple files are written in rapid succession. Because it runs in the background, Claude's response is never blocked.

### Prerequisites: `research/` must be a separate git repo

The hook expects `research/` to be its own git repository with a remote named `origin`. If it isn't, the hook silently exits (no error).

**Setup:**
```bash
cd your-project/research
git init
git remote add origin https://github.com/your-user/your-research-repo.git
git push -u origin main
```

Or clone an existing repo directly:
```bash
git clone https://github.com/your-user/your-research-repo.git research
```

### If `research/` is NOT a separate repo

If you want to keep everything in one repo, the hook won't fire (since the path check requires `/research/` in the file path and the directory won't have its own git remote). In this case, logs are committed to the main repo with your normal git workflow.

### Custom sync command

To override the default sync behavior, set `RESEARCH_SYNC_CMD` in `CLAUDE.md`:

```bash
RESEARCH_SYNC_CMD="bash scripts/my_sync.sh"
```

### Hook registration

The hook is registered in `.claude/settings.local.json`:

```json
"PostToolUse": [
  { "matcher": "Write", "hooks": [{ "type": "command", "command": ".claude/hooks/sync-research-log.sh" }] },
  { "matcher": "Edit",  "hooks": [{ "type": "command", "command": ".claude/hooks/sync-research-log.sh" }] }
]
```

Note: `settings.local.json` is **not** committed to git by default. When copying this pipeline to a new project, you need to re-register the hook in your own `settings.local.json`.
